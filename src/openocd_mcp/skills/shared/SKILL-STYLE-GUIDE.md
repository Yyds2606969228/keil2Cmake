# OpenOCD Skill Style Guide

本文件用于统一 `skills/` 下文档风格、章节结构、边界表达与 YAML 提示词写法。

## 1. 推荐章节顺序

每个主 `SKILL.md` 建议统一采用以下结构：

1. 标题
2. 角色定位
3. 何时使用
4. 标准流程
5. 输出要求
6. 路由规则
7. 共享规范
8. 参考资料

若确有必要增加内容，优先作为上述章节的子项，不建议新增平级大章节。

## 2. 文风要求

- 中文说明为主，术语可保留英文关键词。
- 句式尽量短，动作导向，避免空泛修辞。
- 优先使用清单、编号步骤和可复用模板。
- 结论必须与证据绑定，避免“经验判断式”空结论。
- 能给地址、寄存器、错误码时，不要只给抽象描述。

## 3. 章节编写要求

### 3.1 角色定位
- 写清主责。
- 写清不负责的边界。
- 写清与其他 skill 的关系。

### 3.2 何时使用
- 优先写“触发条件”，不是泛化能力清单。
- 触发条件尽量是现场信号，例如：
	- 连接失败
	- HardFault
	- 需要 ELF/SVD
	- 需要批量回归

### 3.3 标准流程
- 应包含前置检查。
- 应包含失败时的退路。
- 应包含停止条件或交接条件。

### 3.4 输出要求
- 应与 `COMMON-NORMS.md` 中的结构化字段保持一致。
- 必须说明哪些信息是“必填证据”。

### 3.5 路由规则
- 应说明向谁交接。
- 应说明交接时必须带哪些信息。

## 4. 内容边界

- `core`：连接、控制、安全、刷写、恢复
- `triage`：第一现场、快速分流、双通道取证
- `deep-analysis`：栈帧、寄存器、符号、RTOS、多核
- `automation`：脚本、回归、批量验证

严禁一个 skill 吞并全部流程。边界清晰比“功能看起来更多”更重要。

## 5. 引用规则

每个主 `SKILL.md` 应至少引用：
- 1 份共享规范：`../shared/COMMON-NORMS.md`
- 1 份共享风格指南：`../shared/SKILL-STYLE-GUIDE.md`
- 2 份以上本 skill 的关键参考资料

每个 `references/*.md` 建议采用统一模板：
1. 目标
2. 输入
3. 步骤
4. 输出
5. 风险
6. 示例

## 6. YAML 接口风格

- `display_name`：使用 `OpenOCD <Domain>` 格式。
- `short_description`：一句话说明主责，不要堆叠过多名词。
- `default_prompt`：
	- 以 `Use this skill when...` 开头
	- 明确触发条件
	- 明确首选动作
	- 明确输出风格
	- 明确边界或禁止事项

## 7. 写作反模式

以下写法应避免：
- “用于各种调试场景”这类无边界描述
- “可能、也许、大概”却没有证据说明
- 没有失败路径的流程
- 不说明输入前提就直接给步骤
- 只给示例，不给模板字段

## 8. 提交前自检清单

更新 skill 文档前，至少确认：
1. 链接路径正确。
2. 章节顺序统一。
3. 术语一致，例如 `halt`、`watchpoint`、`raw_output`。
4. 高风险动作已经显式标记。
5. 路由规则可执行，不是泛泛建议。
