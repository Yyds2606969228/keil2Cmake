# OpenOCD Skills Common Norms

本文件定义所有 OpenOCD skill 共享的最低执行标准，用于保证行为可审计、可复现、可交接。

## 1. 适用范围

以下规范适用于：
- `openocd-core-operations`
- `openocd-triage-and-capture`
- `openocd-deep-debug-analysis`
- `openocd-automation-validation`

若单个 skill 与本文件存在冲突，以更严格、更安全的约束为准。

## 2. Safety First

1. 默认先读后写，先观察后变更。
2. 未获得用户明确授权时，不执行以下动作：
	- Flash 写入、擦除、解锁
	- 批量内存写入
	- 改写时钟、复位、boot 配置
	- 可能影响业务现场的 `resume`、`reset run`、`mass erase`
3. 会话不稳定、目标状态不明、供电异常时，优先恢复链路与确认边界，不在污染状态下继续推断。
4. 高风险动作必须同时给出：
	- 影响范围
	- 前置条件
	- Dry Run 或替代观测方案
	- 失败后的回退方式

## 3. Evidence First

1. 先采集证据，再给结论。
2. 每条关键证据至少说明四件事：
	- 地址或对象
	- 原始值
	- 上下文
	- 解释
3. 结论必须和证据绑定，禁止只给“可能是”而不说明依据。
4. 若证据不足，应明确写出“不足以定性”的原因，而不是补全想象链条。

## 4. Structured Output

每次 skill 输出应尽量包含以下字段：
- `summary`：一句话概括当前判断
- `current_state`：连接状态、目标状态、关键上下文
- `evidence`：原始寄存器值、地址、日志片段、快照摘要
- `hypothesis`：基于证据的当前推断
- `next_action`：最小下一步动作
- `risk_or_limit`：当前限制或风险
- `raw_output`：无法稳定结构化时保留底层原始输出

若存在失败结果，应额外包含：
- `success`
- `error_code`
- `message`

## 5. Deterministic Flow

1. 优先使用固定顺序：连接 / 冻结 / 取证 / 分析 / 验证 / 清理。
2. 自动化任务必须带超时、退出条件和异常回收路径。
3. 回归用例必须有可重复触发条件与明确的 pass/fail 标准。
4. 对同一问题重复尝试时，应说明：
	- 本次和上次的差异
	- 为什么值得再试
	- 何时停止尝试

## 6. Routing Discipline

1. 超出当前 skill 主责时，立即路由到更合适的 skill。
2. 路由时必须附带交接信息：
	- 已确认事实
	- 尚未确认的问题
	- 已执行动作
	- 未执行且需要审批的动作
3. 不重复做其他 skill 的深度工作，除非只是为闭环做最小补充。

## 7. Cleanup Discipline

1. 退出前清理临时断点、观察点、后台任务和测试工况。
2. 保留 `error_code` 与 `raw_output`，禁止吞错。
3. 可行时执行安全收尾，例如：
	- 释放断点
	- 恢复目标到约定状态
	- 调用 `emergency_stop()`

## 8. 禁止事项

以下行为视为不合规：
- 未确认目标状态就直接写内存或刷写
- 用推测替代证据
- 只给结论，不给地址、值或寄存器依据
- 已进入深度分析却仍停留在泛泛建议
- 回归测试无超时、无统计、无失败样本

## 9. 交付完成标准

一次 skill 响应达到“合格”至少应满足：
1. 边界清晰：当前 skill 在做什么、不做什么。
2. 证据完整：关键判断可回溯。
3. 输出规范：结构化字段清楚。
4. 风险透明：高风险动作已标识。
5. 可交接：下一个 skill 或下一个人能接着执行。
